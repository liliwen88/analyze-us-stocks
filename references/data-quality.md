# 数据质量管控

本文档定义了数据质量的四层管控体系：
1. **时效性管控** — 不同数据类型的使用有效期
2. **来源可靠性评分** — 四级来源可信度
3. **交叉验证规则** — 多源验证要求
4. **数据完整性评级** — 对分析置信度的影响

Agent 在采集数据后、生成结论前，必须执行数据质量检查。

---

## 一、数据时效性 TTL（Time-To-Live）

### 价格与行情数据

| 数据类型 | 盘中时效 | 收盘后时效 | 超时处理 |
|---------|---------|-----------|---------|
| 实时股价 | < 15 分钟 | < 1 天 | 标注"盘中价格，可能有延迟" |
| 日线 OHLCV | — | < 1 个交易日 | 使用前日收盘价，标注日期 |
| 52周高低 | — | < 5 个交易日 | 标注数据日期 |
| 市值 | — | < 5 个交易日 | 使用最新收盘价计算 |
| 成交量 | < 1 小时 | < 1 个交易日 | 标注"基于近期平均成交量" |
| Beta | — | < 60 天 | 标注计算周期 |
| 隐含波动率 (IV) | < 15 分钟 | < 1 天 | 波动剧烈时缩短到盘中 |

### 基本面数据

| 数据类型 | 有效周期 | 超时处理 |
|---------|---------|---------|
| 季度财报数据 | ≤ 100 天（下一季度发布前） | 使用上一份 10-Q/K 数据，标注发布日期 |
| 年度财报数据 | ≤ 370 天 | 标注是基于哪个财年的数据 |
| 管理层指引 | ≤ 100 天 | 采用最新 earnings call 的指引 |
| SEC Filing (10-K/10-Q) | 最近一份 | 标注文件日期 |
| 资产负债表数据 | ≤ 100 天 | 同财报周期 |

### 新闻与事件数据

| 数据类型 | 有效周期 | 超时处理 |
|---------|---------|---------|
| 突发新闻/催化剂 | < 24 小时 | 标注发布时间 |
| 分析师评级变更 | < 30 天 | 标注评级日期和发布机构 |
| 产品发布/召回 | < 7 天（事件期）/ 持续跟踪 | 标注事件日期 |
| 管理层变动 | < 30 天 | 标注变动日期 |
| SEC 8-K 重大事件 | < 5 天 | 标注 filing 日期 |
| 并购传言 | < 48 小时 | 标注"未经证实" |
| 行业/板块分析 | < 30 天 | 标注发布时间 |
| 宏观数据（CPI/就业/GDP） | 发布周期内有效 | 标注发布日期，下一期发布前仍可参考 |

### 情绪与仓位数据

| 数据类型 | 有效周期 | 超时处理 |
|---------|---------|---------|
| 交易所空头持仓 | < 16 天（半月报告） | 标注数据截止日期 |
| 分析师评级/目标价 | < 30 天 | 标注评级日期 |
| 期权未平仓合约 | < 1 个交易日 | 标注数据日期 |
| 13F 机构持仓 | < 110 天（季度+45天） | 标注报告期，提示延迟风险 |
| Form 4 内幕交易 | < 5 个交易日 | 标注交易日期 |
| 社交媒体情绪 | < 4 小时 | 标注分析时间窗口 |

### TTL 使用规则

1. **Agent 在获取数据后**，必须检查数据时间戳与当前时间的差距
2. **超时数据仍可使用**，但必须在报告中标注数据时间并说明可能过时
3. **严重超时**（超过 2 倍 TTL）的数据，应在结论中明确声明："部分数据可能已过时"
4. **无时间戳的数据**，统一标注为"数据时效未知"，降低该数据项的置信度

## 二、来源可靠性评分（四级制）

### A 级：官方/一手来源（可靠性最高）

| 来源类型 | 具体来源 | 适用数据 |
|---------|---------|---------|
| SEC 官方文件 | EDGAR (10-K, 10-Q, 8-K, S-1, Proxy, Form 4) | 财务数据、管理层讨论、风险因素、内部人交易 |
| 公司官方 | IR 官网 earnings release、presentation | 财务业绩、指引 |
| 交易所 | NYSE, Nasdaq 官网 | 交易数据、上市信息 |
| 美国政府 | 美联储、BLS、BEA、Treasury、FRED | 宏观数据、利率、就业、GDP |
| 法院/监管 | 司法部、FTC、SEC 执法、专利局 | 诉讼、监管行动、反垄断 |

**A 级使用规则**：
- 可以单独作为关键论据
- 优先使用 A 级来源替代低级来源的相同信息
- 标注来源时使用"A-EDGAR"、"A-Fed" 等前缀

### B 级：专业金融数据/媒体（可靠性较高）

| 来源类型 | 具体来源 | 适用数据 |
|---------|---------|---------|
| 专业财经通讯社 | Bloomberg, Reuters, Dow Jones Newswires | 实时新闻、市场数据、经济数据 |
| 金融数据平台 | FactSet, S&P Capital IQ, Refinitiv, Morningstar | 行情、估值、基本面、评级 |
| 交易所数据商 | CBOE, OCC (期权), FINRA (债券) | 期权、做空、债券数据 |
| 信用评级机构 | Moody's, S&P, Fitch | 信用评级 |

**B 级使用规则**：
- 可以作为主要论据来源
- 对于实时行情和市场数据，B 级是最佳可用来源
- 如果同一事实 B 级与 A 级冲突，以 A 级为准

### C 级：财经媒体和分析平台（可靠性中等）

| 来源类型 | 具体来源 | 适用数据 |
|---------|---------|---------|
| 主流财经媒体 | CNBC, WSJ, Financial Times, Barron's, MarketWatch, AP | 新闻报道、分析、市场评论 |
| 投资分析平台 | Seeking Alpha, Yahoo Finance, Finviz, TradingView | 数据汇总、社区分析、技术面 |
| 券商研究（公开） | 公开渠道获取的研报摘要 | 分析师观点 |

**C 级使用规则**：
- 可用于背景信息和辅助判断
- 关键论据建议与 A/B 级来源交叉验证
- Seeking Alpha 等平台上的个人分析需额外谨慎
- WSJ 和 FT 的原创新闻可信度高于普通财经网站

### D 级：非正式来源（可靠性最低）

| 来源类型 | 具体来源 | 适用数据 |
|---------|---------|---------|
| 社交媒体 | Reddit, X/Twitter, StockTwits, Discord | 市场情绪（仅作为情绪指标） |
| 论坛/留言板 | Yahoo Finance comments, 雪球 | 散户讨论 |
| 匿名来源 | 匿名分析师、未署名报告 | 不应作为论据 |
| AI 生成内容 | ChatGPT、其他 LLM 生成的金融分析 | 不应作为事实来源 |

**D 级使用规则**：
- **严禁作为事实论据的唯一来源**
- 社交媒体情绪数据（如 Reddit WSB 讨论量）可作为情绪指标，但需标注"D-社交情绪"
- D 级来源的信息如欲使用，必须先向 A/B 级来源验证

### 来源选择决策树

```
关键数据点
  ├─ 有 A 级来源？→ 使用 A 级，标注 A-[来源]
  ├─ 无 A，有 B 级来源？→ 使用 B 级，标注 B-[来源]
  ├─ 无 A/B，有 C 级来源？→ 使用 C 级，标注 C-[来源]，并注明"需更高可靠性验证"
  └─ 仅 D 级来源？→ 不纳入关键论据，或标注"未经可靠来源验证"
```

## 三、交叉验证规则

### 3.1 强制交叉验证（Must Cross-Validate）

以下数据类型必须至少从 2 个独立来源交叉验证：

| 数据项 | 验证要求 | 说明 |
|--------|---------|------|
| 最新季度营收/EPS | ≥2 来源 | 交易所与 IR 官网对比 |
| 管理层指引 | ≥2 来源 | Earnings call + IR release |
| 重大新闻事件 | ≥2 来源 | 不同媒体，排除转载 |
| 分析师评级变更 | ≥2 来源 | 确认评级、目标价和日期 |
| 内幕交易数据 | ≥2 来源 | SEC Form 4 + 数据聚合平台 |
| 空头持仓数据 | ≥2 来源 | 交易所 + 数据平台 |
| 并购/分拆公告 | ≥2 来源 | 公司 IR + 财经媒体 |

### 3.2 推荐交叉验证（Should Cross-Validate）

以下数据类型建议从 2 个来源验证：

| 数据项 | 验证要求 |
|--------|---------|
| 最新股价/行情 | 行情提供商 + 交易所 |
| 估值倍数 | 数据平台 + 手工验证（市值/盈利） |
| 机构持仓变动 | 13F 原始数据 + 聚合平台 |
| 同行估值数据 | ≥2 个数据平台 |

### 3.3 来源冲突处理

当两个以上可靠来源对同一数据给出不同值时：

1. **确定差异范围**：差异 < 5% 通常可忽略
2. **按来源等级选择**：A > B > C > D
3. **标注冲突**：在报告中注明"数据存在分歧：[来源1]显示 X，[来源2]显示 Y"
4. **选择更可靠来源**：解释选择理由
5. **敏感度处理**：如数据分歧影响结论，在报告中展示两种情形

### 3.4 来源独立性检查

以下情况不算独立来源，只能算作 1 个来源：

- 同一原始数据的转载
- 同一新闻通讯社的文章在不同媒体分发
- 同一数据提供商的同一数据在不同平台展示
- AI 生成的汇总内容（应追溯到原始来源）

## 四、数据完整性评级

在完成数据采集后，Agent 应对数据完整性进行评级：

### 完整（Complete）

- **定义**：所有必查数据项均已获取且时效在 TTL 内
- **对置信度的影响**：不降低
- **报告标注**：可能需要也可能不需要标注（数据完整是默认期望）

### 部分缺失（Partial）

- **定义**：1-3 个必查数据项缺失或超时
- **对置信度的影响**：置信度降 1 级（高→中、中→低）
- **报告标注**：在"来源和局限"中列出缺失项及原因
- **例子**："最新季度财报尚未发布，使用上季度数据"；"内幕交易数据因 SEC EDGAR 不可用而未获取"

### 严重缺失（Severe）

- **定义**：4+ 个必查数据项缺失或核心数据（如最新财报、最新行情）严重超时
- **对置信度的影响**：置信度最高为"低"
- **报告标注**：在结论部分开头明确声明数据局限性
- **处理建议**：建议用户等待数据可获取后再做决策；推迟给出明确操作建议

## 五、数据质量检查清单

Agent 在完成数据采集后、进入分析前，必须执行以下检查：

```
□ 1. 行情数据时间戳是否在 TTL 内？
□ 2. 财报数据是否为最新季度/年度？
□ 3. 关键新闻是否标注了发布时间？
□ 4. 每个关键数据点的来源是否标注了可靠性等级（A/B/C/D）？
□ 5. 强制交叉验证的数据项是否有 ≥2 个独立来源？
□ 6. 是否有 D 级来源被误用为关键论据？
□ 7. 数据完整性评级是否已完成？
□ 8. 冲突数据是否已标注并解释了选择？
□ 9. 超时数据是否已在报告中标注？
□ 10. 无法获取的数据是否在局限性部分列出？
```

## 六、与 Agent 工作流的集成

### 在数据采集阶段（SKILL.md 步骤 2）

Agent 在采集每项数据时，应同步记录：
- 数据值 + 单位
- 来源 URL/名称
- 来源可靠性等级 (A/B/C/D)
- 数据时间戳

### 在交叉验证阶段（SKILL.md 步骤 3.5）

Agent 应系统性检查：
- 强制交叉验证项是否满足
- 来源冲突是否存在
- 数据完整性评级

### 在报告生成阶段（SKILL.md 步骤 5）

报告"来源和局限"部分应包含：
- 数据完整性评级
- 缺失数据列表
- 超时数据列表
- 来源可靠性分布（按 A/B/C/D 统计）
- 任何值得注意的来源冲突

### 对概率/置信度的影响

| 数据完整性 | 来源可靠性分布 | 最高置信度 |
|-----------|-------------|----------|
| 完整 | A/B 级主导 | 高 |
| 完整 | C 级主导 | 中（可上调至高若有多源验证） |
| 部分缺失 | A/B 级主导 | 中 |
| 部分缺失 | C 级主导 | 低 |
| 严重缺失 | 任何 | 低 |
